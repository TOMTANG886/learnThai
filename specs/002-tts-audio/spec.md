# Feature Specification: TTS Audio Generation Service

**Feature Branch**: `002-tts-audio`  
**Created**: 2026-03-08  
**Status**: Draft  
**Input**: User description: "Implement an isolated TTS/audio-generation service module that matches Principle III: input = text + voice/lang config, output = audio file path (or blob) and content-hash"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Build Pipeline Audio Generation (Priority: P1)

A developer runs the build pipeline with the `--generate-audio` flag. The system fetches Thai text content from Google Sheets, generates audio files for each post using a TTS provider, and caches the audio files by content hash in `public/assets/audio/`. Subsequent builds with identical content reuse cached files without making redundant API calls.

**Why this priority**: This is the core value proposition—automated audio generation during the build process. Without this, the feature provides no value.

**Independent Test**: Run `npm run build -- --generate-audio` with a sample Sheet containing Thai text. Verify audio files are created in `public/assets/audio/` with hash-based filenames. Run again with identical Sheet data and confirm no new API calls are made (cache hit).

**Acceptance Scenarios**:

1. **Given** a Google Sheet with Thai text in the `body_markdown` column, **When** the build runs with `--generate-audio`, **Then** audio files are created for each post with hash-based filenames
2. **Given** audio files already exist for specific content hashes, **When** the build runs again with identical Sheet data, **Then** no new TTS API calls are made and existing files are reused
3. **Given** Sheet content changes for one post, **When** the build runs, **Then** only the changed post triggers a new TTS API call

---

### User Story 2 - Local Provider for Development (Priority: P2)

A developer without production TTS credentials runs audio generation locally using a development provider mode (including mock generation and script-based local generation). The workflow completes successfully for local validation and iteration.

**Why this priority**: Enables local development and testing without production keys, reducing contributor setup friction.

**Independent Test**: Run build with local provider mode enabled and no production credentials. Verify local audio outputs are created and build completes successfully.

**Acceptance Scenarios**:

1. **Given** local provider mode is enabled, **When** audio generation is invoked, **Then** local audio artifacts are created in the output directory
2. **Given** no production TTS credentials configured, **When** local provider mode is used, **Then** build completes without credential errors
3. **Given** unchanged text and voice/language settings, **When** local generation reruns, **Then** previously generated artifacts are reused by content hash

---

### User Story 3 - HTTP Provider Integration (Priority: P3)

A developer configures an HTTP-based TTS provider (e.g., Google Cloud TTS, AWS Polly) via environment variables. The system calls the provider API with text, voice, and language parameters, receives binary audio data, and saves it to disk with appropriate error handling.

**Why this priority**: Provides real production audio generation capability. Lower priority because mock provider enables development and testing.

**Independent Test**: Configure `TTS_PROVIDER=http`, `TTS_API_URL`, and `TTS_API_KEY`. Run build and verify real audio files are generated by calling the external API.

**Acceptance Scenarios**:

1. **Given** valid HTTP provider credentials, **When** audio generation is invoked, **Then** API receives POST request with text, voice, and lang parameters
2. **Given** API returns audio binary data, **When** response is received, **Then** data is written to disk as .mp3 file
3. **Given** API returns error status, **When** generation fails, **Then** build fails with actionable error message containing status code
4. **Given** HTTP provider integration is not enabled, **When** generation runs, **Then** the system uses Local Provider mode instead

---

### User Story 4 - Audio Playback Component (Priority: P1)

A visitor opens a blog post page in their browser. The page displays an audio player component that loads and plays the generated Thai audio file. The player provides basic controls (play/pause, progress bar) and handles loading states and errors gracefully.

**Why this priority**: Audio generation is useless without playback capability. This is essential for the blog's user-facing functionality.

**Independent Test**: Open any blog post page at `/worksheets/<slug>` in a browser. Verify an audio player is visible, clicking play loads and plays the audio file, and progress updates in real-time.

**Acceptance Scenarios**:

1. **Given** a post page with an audio file path, **When** the page loads, **Then** an audio player component is rendered
2. **Given** the audio player is rendered, **When** user clicks play, **Then** audio loads and plays with visible progress indicator
3. **Given** audio is playing, **When** user clicks pause, **Then** playback stops and can be resumed
4. **Given** audio file is missing or fails to load, **When** player attempts to load, **Then** user-friendly error message is displayed

### Edge Cases

- What happens when the audio output directory doesn't exist? → System must create it automatically.
- How does system handle provider rate limits or temporary outages? → Build should fail with clear, actionable retry guidance.
- What happens when identical text has different voice/lang configs? → Different hashes are generated and separate artifacts are created.
- How does system handle empty or whitespace-only text? → Skip generation for that item and record a clear warning.
- What happens when disk is full during file write? → Build fails with a clear error and no partial success is reported.
- What happens when a page references a missing audio file? → Page shows fallback state and remains otherwise usable.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Service module MUST expose `generateAudioForPost(text, slug, opts)` function accepting text string, slug identifier, and TtsOptions config
- **FR-002**: Service module MUST expose `hashTextForAudio(text, opts)` function returning deterministic SHA-256 hash from text and config
- **FR-003**: Service module MUST return `{ path: string, hash: string }` where path is public URL and hash is content identifier
- **FR-004**: Service module MUST cache generated audio files in `public/assets/audio/` directory with hash-based filenames (format: `{hash}.mp3`)
- **FR-005**: Service module MUST skip API calls when audio file with matching hash already exists on disk (idempotency)
- **FR-006**: Service module MUST support configurable TTS provider via `TTS_PROVIDER` environment variable
- **FR-007**: Service module MUST provide a local development provider mode that can run without production credentials
- **FR-008**: Service module MUST implement an 'http' provider that calls external TTS API with POST requests
- **FR-009**: HTTP provider MUST read `TTS_API_URL` and `TTS_API_KEY` from environment variables
- **FR-010**: HTTP provider MUST send JSON payload with `text`, `voice`, `lang`, and `format` parameters
- **FR-011**: HTTP provider MUST handle binary audio response data and write to disk
- **FR-012**: Service module MUST create audio output directory if it doesn't exist
- **FR-013**: Service module MUST throw actionable errors with status codes when external API fails
- **FR-014**: Service module MUST NOT log or expose API credentials in error messages or console output
- **FR-015**: Service module MUST be testable without live external API (via local development provider mode)
- **FR-016**: Local provider mode MUST support developer-script based generation for local validation workflows
- **FR-017**: If HTTP provider integration is not selected or not configured, the system MUST run generation using Local Provider mode
- **FR-018**: Audio player component MUST accept an `audioPath` prop pointing to the audio file URL
- **FR-019**: Audio player component MUST render HTML5 `<audio>` element with controls (play, pause, progress)
- **FR-020**: Audio player component MUST display loading state while audio file is being fetched
- **FR-021**: Audio player component MUST display error state when audio file fails to load
- **FR-022**: Audio player component MUST be implemented as a reusable component
- **FR-023**: Audio player component MUST work with static site generation (no runtime dependencies)
- **FR-024**: Audio player component MUST be accessible (keyboard navigation, ARIA labels)

### Key Entities

- **TtsOptions**: Configuration object containing optional `voice` (string), `lang` (string), and `format` ('mp3' | 'ogg') parameters
- **AudioFile**: Generated audio artifact with `path` (public URL string) and `hash` (SHA-256 content identifier string)
- **TtsProvider**: Service adapter implementing audio generation strategy (local provider mode and production provider mode, extensible for additional providers)
- **AudioPlayerProps**: Component props containing `audioPath` (string), optional `title` (string), and optional `className` (string) for styling

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Build pipeline completes successfully with audio generation enabled on first run
- **SC-002**: Subsequent builds with identical Sheet data complete 50%+ faster due to cache hits (no redundant API calls)
- **SC-003**: Local development builds complete successfully without production credentials using local provider mode
- **SC-004**: Audio files are deterministically named by content hash—same text+config always produces same filename
- **SC-005**: Build fails fast with actionable error message when TTS API is unreachable or credentials are invalid
- **SC-006**: Zero API credentials appear in build logs, error messages, or generated static files
- **SC-007**: Service module passes unit tests without requiring live external API
- **SC-008**: Blog post pages display audio player component that loads and plays audio files
- **SC-009**: Audio player provides functional play/pause controls and visual progress feedback
- **SC-010**: Audio player handles loading and error states gracefully without breaking page layout
- **SC-011**: In runs where HTTP provider mode is not enabled, 100% of generated artifacts are produced via Local Provider mode

## Assumptions

- Node.js 18+ runtime is available (for native `fetch` API support in HTTP provider)
- Google Sheets API integration already exists in the project (per constitution Principle II)
- Build pipeline orchestrator (`build-from-sheets.ts`) exists and can be extended with audio generation calls
- Audio files up to 5MB each are acceptable for static hosting
- MP3 format is sufficient for initial implementation (OGG support is optional)
- TTS API providers return audio binary data in response body (standard REST pattern)
- Project uses environment variables for secrets (per constitution External Service Contracts)

## Out of Scope

- Real-time audio generation (all generation happens at build time per constitution Principle I)
- Advanced audio player features (speed control, volume, download button, playlist)
- Audio transcoding/format conversion beyond provider output
- Retry logic with exponential backoff for API failures (fail-fast is acceptable for initial implementation)
- Audio quality validation or post-processing (e.g., normalization, compression)
- Multiple TTS provider fallback chains
- Audio streaming or chunked generation for very long text
- Mobile-optimized player UI (basic HTML5 controls are sufficient initially)
- Audio waveform visualization or interactive timeline
