# Research: TTS Audio Generation & Playback

**Feature**: 002-tts-audio  
**Created**: 2026-03-08  
**Phase**: 0 (Research & Decision Making)

## Research Questions Resolved

### Q1: How should audio player component integrate with Next.js static pages?

**Decision**: Use standard HTML5 `<audio>` element wrapped in a React component

**Rationale**:
- Next.js static export fully supports HTML5 audio elements
- No additional dependencies required (browser native API)
- Works consistently across all modern browsers
- Accessible by default with proper ARIA attributes
- Lightweight—no bloated audio player libraries needed

**Alternatives Considered**:
- **Third-party audio libraries** (react-h5-audio-player, Plyr): Rejected due to unnecessary complexity and bundle size increase; Principle V (Simplicity) dictates minimal client-side JS
- **Custom Web Audio API implementation**: Rejected as over-engineering; HTML5 audio provides all needed functionality
- **Server-side audio streaming**: Rejected as it violates Principle I (Static-First); all audio files are pre-generated static assets

**Implementation Pattern**:
```tsx
// web/src/components/AudioPlayer.tsx
export function AudioPlayer({ audioPath, title }: AudioPlayerProps) {
  return (
    <audio controls preload="metadata" aria-label={`Audio for ${title}`}>
      <source src={audioPath} type="audio/mpeg" />
      Your browser does not support audio playback.
    </audio>
  );
}
```

### Q2: Where should audio player component be located in the project structure?

**Decision**: Create `web/src/components/AudioPlayer.tsx` for the React component

**Rationale**:
- Follows existing Next.js project structure conventions
- Co-located with other UI components (if any exist)
- Easy to import from page files: `import { AudioPlayer } from '@/components/AudioPlayer'`
- Testable independently from pages

**Alternatives Considered**:
- **Inline in page templates**: Rejected to maintain component reusability
- **In lib/ directory**: Rejected as lib/ is for non-UI logic (services, utilities); components belong in components/

---

### Q3: How should post pages access audio file paths?

**Decision**: Pass audio path via page props during static generation (getStaticProps)

**Rationale**:
- Aligns with Next.js static site generation pattern
- Audio file path is already in build manifest (generated by build-from-sheets.ts)
- No runtime API calls needed—data embedded at build time (Principle I)

**Implementation Pattern**:
```tsx
// web/pages/worksheets/[worksheet].tsx
export async function getStaticProps({ params }) {
  const post = await loadPostData(params.worksheet);
  return {
    props: {
      ...post,
      audioPath: post.audio || null, // from manifest
    },
  };
}

export default function WorksheetPage({ audioPath, title, ...post }) {
  return (
    <div>
      <h1>{title}</h1>
      {audioPath && <AudioPlayer audioPath={audioPath} title={title} />}
      {/* rest of content */}
    </div>
  );
}
```

### Q4: How should the component handle missing or failed audio loads?

**Decision**: Show fallback UI on error; gracefully degrade to text-only content

**Rationale**:
- Audio is enhancement, not requirement (progressive enhancement)
- Page should remain functional even if audio fails to load
- User should see clear feedback, not broken player

**Implementation Pattern**:
```tsx
export function AudioPlayer({ audioPath, title }: AudioPlayerProps) {
  const [error, setError] = useState(false);
  
  if (error) {
    return <p className="audio-error">Audio unavailable for this post.</p>;
  }
  
  return (
    <audio
      controls
      preload="metadata"
      onError={() => setError(true)}
      aria-label={`Audio for ${title}`}
    >
      <source src={audioPath} type="audio/mpeg" />
      Your browser does not support audio playback.
    </audio>
  );
}
```

### Q5: Should audio player support additional features (speed, download, etc.)?

**Decision**: No—use browser default controls only for initial implementation

**Rationale**:
- Principle V (Simplicity): Don't add features without proven user need
- Browser native controls provide play/pause/seek/volume out of the box
- Adding custom controls increases bundle size and complexity
- Can add custom controls in future iteration if user feedback demands it

**Future Considerations** (not in scope):
- Playback speed control (1x, 1.5x, 2x)
- Download button for offline use
- Keyboard shortcuts (j/k for skip back/forward)
- Mobile-optimized UI

## Technology Stack Decisions

### Audio Format

**Decision**: MP3 (MPEG-1 Audio Layer 3)

**Rationale**:
- Universal browser support (all modern browsers, all devices)
- Good compression ratio (smaller file sizes than WAV, comparable to OGG)
- TTS providers commonly output MP3 format
- Already decided in spec (FR-011)

**Compression Settings** (for future TTS provider config):
- Bitrate: 64kbps for speech (sufficient quality, small size)
- Sample rate: 22.05kHz or 24kHz (standard for TTS output)

---

### Component Library

**Decision**: No component library—plain React + HTML5 audio

**Rationale**:
- HTML5 audio API is sufficient for requirements
- No need for styled components or CSS-in-JS libraries
- Keep bundle size minimal (Principle I: minimal client-side JS)
- Use project's existing CSS/styling approach

---

### State Management

**Decision**: No global state—component-local useState only

**Rationale**:
- Each audio player is independent per post
- No need to coordinate playback across multiple players
- No need for Redux/Zustand/Jotai overhead
- Simple useState for error handling is sufficient

## Constitution Compliance Check

✅ **Principle I (Static-First)**: Audio files are pre-generated at build time; player uses static assets  
✅ **Principle II (Sheets SOT)**: Audio paths come from manifest generated from Google Sheets data  
✅ **Principle III (Isolated Services)**: Player component is isolated from TTS service; clear interface  
✅ **Principle IV (Idempotent Pipeline)**: Player consumes output of idempotent audio generation  
✅ **Principle V (Simplicity)**: Minimal client-side JS; no unnecessary libraries or abstractions
